# AgentVault Roadmap

## 0. Mission

AgentVault provides structured, bounded, verifiable agent-to-agent coordination
with privacy-preserving outputs.

AgentVault is the "everyday lane" sibling to hardened VCAV-H. It prioritises:

- Mechanically enforceable bounded outputs
- Verifiable execution via receipts
- Low operational complexity
- Extensibility without protocol rewrites

---

# I. Design Principles (Non-Negotiable)

These principles guide all roadmap decisions.

## 1. Bounded Disclosure is Mechanical, Not Normative

Privacy is not enforced by prompt instructions.

It is enforced by:

- Structured output schemas
- Deterministic validation
- Explicit entropy constraints
- Policy rules (where applicable)

Prompt programs are guidance only.

## 2. All Enforcement-Relevant Artefacts Are First-Class

If it affects what can come out of the vault, it must be:

- Explicit
- Versioned
- Content-addressed (hashable)
- Bindable into receipts

This includes:

- Contracts
- Output schemas
- Prompt programs
- Model profiles
- Policy bundles (future)
- Guardian configuration (future)

## 3. Contracts Reference Artefacts, Not Behaviour

No hidden logic such as:

```
if contract_type == "COMPATIBILITY" then do X
```

Instead:

- Contract selects schema
- Contract selects prompt program
- Contract selects model profile
- Contract selects policy bundle (future)

Behaviour derives from referenced artefacts.

## 4. Enforcement is Deterministic and Explicitly Classified

Any rule that affects output validity must:

- Be reproducible offline
- Be reflected in receipts (if release-gating)
- Not depend on model interpretation

Enforcement rules must be explicitly classified as either:

- **GATE** (release-blocking)
- **ADVISORY** (non-blocking, diagnostic only)

The classification must be visible in code or configuration (e.g. in PolicyBundle
metadata) to prevent silent drift over time.

## 5. Receipts Bind the Consent Boundary

Receipts must bind enough information to answer:

- What contract was executed?
- What schema was enforced?
- What model profile was used?
- What prompt program was used?
- What policy bundle was enforced? (future)
- What runtime built this?

Receipts should never imply guarantees stronger than what the system actually
enforces.

## 6. Novel Contracts Are Allowed. Novel Powers Are Not.

Users may define new contracts and schemas.

They may not introduce new classes of enforcement without upgrading the guardian
or relay.

This preserves extensibility without uncontrolled surface expansion.

# Phase 1 — Immediate Hardening

Scope: Close red-team leak class and remove major footguns.

## 1. All-Enum COMPATIBILITY Schema (v2) — DONE

*Completed: PR #20, merged 2026-02-26.*

- Removed free-text `overlap_summary`
- Introduced orthogonal enum dimensions (`thesis_fit`, `size_fit`, `stage_fit`,
  `confidence`, `primary_reasons`, `blocking_reasons`, `next_step`)
- Added bounded reason arrays (`uniqueItems: true`, `maxItems: 3` / `2`)
- Set `additionalProperties: false`
- Bumped schema version and contract hash
- Entropy budget: 32 bits (advisory); actual: ~22 bits

## 2. Relay-Side Deterministic Digit/Currency Guard — DONE

*Completed: relay-side GATE rule using Unicode categories Nd (numeric) and Sc
(currency symbols). Scoped to `vcav_e_compatibility_signal_v2` via hardcoded
schema ID. Constant-shape error response (`OUTPUT_POLICY_VIOLATION`). No
offending content in logs. Schema-first invariant regression-proofed by unit test.*

Scoped initially to COMPATIBILITY v2:

- After schema validation, reject if any string value contains:
  - Unicode numeric characters (category Nd — covers ASCII, fullwidth, Arabic-Indic)
  - Unicode currency symbols (category Sc — covers £, $, €, ¥, ₹, etc.)
- Abort with constant-shape error

This is a **GATE** rule.

Purpose: Defense-in-depth backstop / schema regression detector. The primary
privacy control is the all-enum schema with `additionalProperties: false`.
This guard fires only if the schema is misconfigured, weakened, or a provider
structured-output bug bypasses enum constraints.

## 3. Model Profile Immutability

Add a lockfile (`model_profiles.lock`) that pins each profile ID to its
expected content hash. Relay refuses to start if any profile's computed hash
does not match the lockfile entry.

This is simpler than hash-named files (Option A considered, rejected — adds
indirection for no enforcement gain over a lockfile, and breaks human
readability of profile directories). The lockfile is regenerated by an
explicit CLI command (`relay lock-profiles`), never silently.

Goal: Eliminate silent profile mutation.

**Current state:** Partially done. Content hash is computed at runtime
(`ModelProfile::content_hash()`) and bound into receipts. But profile files
are still ID-named (`api-claude-sonnet-v1.json`), so a file can be mutated
between runs without detection until receipt comparison. The gap is relay-side
startup validation.

## 4. Honest Runtime Hashing

Replace `SHA256("version-string")` with:

- Git commit SHA embedded at build time, or
- Docker image digest

Receipts must not claim unverifiable runtime provenance.

**Current state:** Not done. Relay uses `SHA256("vcav-e-relay-v0.2.0-bilateral")`
as `runtime_hash`, `model_weights_hash`, and `inference_config_hash`. This is a
placeholder, not a real build artefact hash.

## 5. Phase 1 Exit Criteria

Phase 1 is complete when:

**Mechanical (must pass in CI):**
- Relay-side digit/currency guard has deterministic unit tests
- Schema validation rejects `additionalProperties`
- Model profile startup hash check passes

**Evidential (documented, not CI-gated):**
- Red-team suite passes across providers (20/20 — done, see
  `docs/red-team-evaluation-notes.md`)
- Accumulation tests operate purely on enum output (done)
- No numeric or currency tokens appear in SAFE contract output (done post-hoc,
  needs relay-side enforcement)

# Phase 2 — Artefact Discipline (Short-Term)

Scope: Align implementation with design principles without major architecture
shift.

## 6. Output Schema as Standalone Artefact

Currently:
- Schema embedded inside contract
- Referenced by `output_schema_id` (human-readable string)

Target:
- Standalone schema artefact
- Referenced by `output_schema_hash`
- Schema registry (simple JSON or directory-based)

Receipt includes `output_schema_hash`.

**Current state:** Not started. Schema is embedded in contract and bound
indirectly via contract hash. No separate content-addressing.

## 7. Policy Bundle v1 (Minimal)

Introduce first-class PolicyBundle artefact.

Example structure:
- `allowed_schema_hashes`
- `allowed_model_profile_hashes`
- `reject_patterns`
- `entropy_constraints` (including budget governance — see below)
- `max_tokens`
- `provider_allowlist`
- `enforcement_classification` (GATE vs ADVISORY per rule)

The `entropy_constraints` section must define how entropy budgets are set and
reviewed. Currently the 32-bit advisory budget for COMPATIBILITY v2 is a
hardcoded constant with no review trigger. The PolicyBundle should declare:
- Budget value and classification (GATE vs ADVISORY)
- Review trigger (e.g. "review if actual exceeds 80% of budget")
- Tightening policy (e.g. "reduce after schema change if actual headroom > 50%")

This prevents entropy budgets from silently drifting as schemas evolve.

Enforced in relay (no Guardian rewrite yet).

Receipt binds `policy_bundle_hash`.

**Dependency:** Requires item 6 (schema-as-standalone-artefact) to be done
first, since the policy bundle references schema hashes.

**Current state:** Not started. No PolicyBundle type in codebase. Policy is
currently represented by flat fields (`purpose_code`, `entropy_budget_bits`,
`guardian_policy_hash`).

## 8. Capability Declaration in Policy

Policy bundle declares required capabilities:

- `schema_validation`
- `entropy_tracking`
- `reject_patterns`
- `constant_shape`

Relay must assert it supports declared capabilities.

This prevents silent semantic drift.

## 9. Schema Versioning & Migration Policy

Explicitly define:

- Schema hashes are immutable
- Contracts are bound to specific schema hashes
- No in-place migration of existing contracts
- Parties must negotiate schema version out-of-band

Backwards compatibility is achieved by coexistence, not mutation.

# Phase 2b — OpenClaw Integration (Parallel Track)

Scope: First real peer-to-peer vault session between independent agents.

Per `docs/plans/open_claw_agent_vault_mcp_port_brief_v_1.md`.

This track runs in parallel with Phase 2 artefact discipline. It depends on
Phase 1 being complete (v2 schema hardened) but does not depend on Phase 2.

## 10. OpenClaw Skill + mcporter Integration

- Create OpenClaw Skill (`~/.openclaw/skills/agentvault/SKILL.md`)
- Validate mcporter can invoke all AgentVault MCP tools
- Test AFAL direct transport between two VPS instances
- No native plugin port initially (mcporter indirection only)

The skill teaches the agent how to use AgentVault tools, encodes the correct
lifecycle, and prevents protocol instructions from appearing in user prompts.

## 11. VPS Runbook + First Live Session

- Pre-flight checklist on both hosts (OpenClaw, mcporter, MCP server)
- Natural-language prompt → full vault session → verifiable receipts
- No manual coordination, no prompt-based protocol hints
- Capture: receipts, session pointers, logs

Success criteria:
- Receipts verifiable
- No out-of-band coordination
- Session fully completed via tool-mediated flow

## 12. Native OpenClaw Plugin (Optional, Phase 4+)

A TypeScript plugin could:
- Provide native tools instead of mcporter indirection
- Expose configuration via `openclaw.plugin.json`
- Add UI hints

This increases integration tightness but expands trusted surface.

Not required for first peer-to-peer test. Only pursue if mcporter path proves
limiting.

# Phase 3 — Registry & Versioning Infrastructure

Scope: Prepare for ecosystem-scale usage.

## 13. Formal Registries

Introduce registries for:

- Schemas
- Policy bundles
- Model profiles
- Prompt programs

Properties:
- Append-only
- Content-addressed
- ID separate from hash
- Deprecation without deletion

Trust model for registry discovery and key distribution is explicitly out of
scope for AgentVault (see Out of Scope section).

## 14. Receipt Completeness Improvements

Add explicit binding of:

- `output_schema_hash`
- `policy_bundle_hash`
- `runtime_build_hash` (real)
- `model_profile_hash` (already present)
- `prompt_program_hash` (already present)

Receipts must reflect actual enforced artefacts.

**Current state:** Receipts bind 24+ fields including contract hash, prompt
template hash, model profile hash, model identity, entropy, and budget usage.
Missing: `output_schema_hash`, `policy_bundle_hash`, real `runtime_build_hash`.

## 15. Extended Accumulation Experiment (N=20-100)

Red team testing at N=3 showed no accumulation (see decision log, 2026-02-26).
This is a negative result at low N, not a proof of resistance.

Run extended accumulation experiments:
- N=20, N=50, N=100 sessions per (scenario, provider) pair
- Scenarios 06 (naive), 07 (strategic), 08 (expert)
- Measure: signal entropy per session, cross-session narrowing, reconstruction
  accuracy vs random baseline
- Feed results back into entropy budget governance (item 7)

If narrowing is detected at higher N, the response is to tighten the entropy
budget or reduce enum cardinality — not to change the protocol.

**Current state:** Not started. `accumulate.sh` supports multi-session
analysis but has only been run at N=3. Infrastructure exists; experiment time
is the bottleneck.

## 16. Persistent Receipt Storage (Optional)

Currently relay is stateless.

Optional feature:
- Append-only receipt log
- Useful for audit deployments

Not required for core AgentVault UX.

# Phase 4 — Extensibility & Custom Contracts

Scope: Enable safe bespoke contracts.

## 17. Custom Contract Builder (Advanced UX)

Allow users to:

- Define new output schemas
- Define policy bundles
- Generate content hashes
- Publish to local registry namespace

All within existing capability boundaries.

## 18. SAFE vs RICH Contract Variants

For COMPATIBILITY and similar:

**SAFE:**
- Enum-only
- No free text
- Deterministic policy gate
- Strict entropy bounds

**RICH** (only allowed once PolicyBundle v1 exists):
- May include bounded free text
- Must define explicit entropy upper bound
- Must define `maxLength` and/or token constraints
- Must include deterministic reject patterns
- Must clearly declare reduced privacy guarantees

SAFE is default.

RICH contracts must be explicitly labeled and not silently substituted.

# III. Out of Scope (For AgentVault Lane)

These belong to VCAV-H or a future hardened lane:

- Hardware enclave attestation
- Threshold attestation
- Multi-operator consensus
- Cryptographic model execution proofs
- Canonical registry key distribution and trust bootstrap

AgentVault remains software-attested, relay-based.
